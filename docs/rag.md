# RAG — похожие вакансии, семантический поиск, Q&A

RAG-слой даёт три вещи поверх корпуса вакансий:

1. **Похожие вакансии** — блок на странице вакансии (`/vacancy/{id}`), retrieval по вектору.
2. **Семантический поиск** — `/search`, запрос на естественном языке → ранжированные вакансии.
3. **Q&A (полный RAG)** — `/search` → «Спросить»: вопрос → находим релевантные вакансии → LLM отвечает, ссылаясь на них `[#id]`.

## Опциональность

RAG — **необязательная** фича. Включается отдельным extra:

```bash
uv sync --extra rag        # ставит sqlite-vec
ollama pull nomic-embed-text   # модель эмбеддингов (~270 MB)
```

Без extra `rag` (`sqlite-vec` не установлен или не грузится в данной сборке sqlite) — `rag.is_available()` возвращает False, и:
- блок «Похожие вакансии» на вакансии скрыт;
- `/search` показывает подсказку «RAG не включён»;
- `/api/rag/search` и `/api/rag/ask` отвечают `{"ok": false, "reason": "rag_disabled"}`;
- cron-джоб `embed_vacancies` — no-op (`skipped: rag_disabled`).

Остальное приложение работает как обычно. Детект кешируется (`app/llm/rag.py::is_available`).

## Как работает

| Слой | Где | Что |
|---|---|---|
| Эмбеддинги | `llm_client.embed()` → Ollama `/api/embed` | `nomic-embed-text`, dim 768 |
| Хранение вектора | `vec_vacancies` (sqlite-vec vec0, `distance_metric=cosine`) | создаётся лениво |
| Мета/coverage/dedup | `vacancy_embeddings` (обычная таблица) | `model`, `dim`, `source_hash` |
| Индексация | джоб `embed_vacancies` (30 мин) + кнопка на `/search` | эмбедит вакансии без вектора |
| Похожие | `rag.similar(vid)` | KNN по вектору вакансии |
| Поиск | `rag.semantic_search(q)` | эмбеддинг запроса → KNN |
| Q&A | `rag.ask(q)` | retrieval + `llm_client.generate` с ссылками `[#id]` |

Каждый вызов эмбеддинга/генерации логируется в `llm_runs` (`task_kind` = `embed` / `rag_answer`) — видно на `/llm-logs`.

## Загрузка расширения

`sqlite-vec` — loadable extension. `rag.load_vec(conn)` грузит его в каждое aiosqlite-соединение, которое обращается к vec0 (идемпотентно — повторный вызов на том же соединении ничего не делает). Общий `get_db()` не трогается, чтобы не-RAG код не платил оверхед на загрузку.

## Индексация корпуса

- Автоматически: cron `embed_vacancies` (каждые 30 мин) эмбедит до 25 вакансий без вектора.
- Вручную: кнопка «⚙️ Индексировать корпус» на `/search` (дёргает `embed_vacancies` через run-now). Покрытие «проиндексировано N из M» — там же.

## Ограничения

- **Размерность фиксирована** (768 под `nomic-embed-text`). Смена embed-модели с другим dim требует пересоздать `vec_vacancies` (удалить таблицу — она создастся заново при следующей индексации).
- Поиск в памяти sqlite-vec — нормально для корпуса до нескольких десятков тысяч вакансий.
- Демо-снапшот (`make snapshot`) использует детерминированные псевдо-эмбеддинги (без Ollama), чтобы «похожие» и поиск рендерились; в реальном режиме вектора считает Ollama.

## Тесты

RAG-тесты помечены `pytest.importorskip("sqlite_vec")` — скипаются, если extra не установлен. В CI (`uv sync --all-extras --dev`) extra стоит, поэтому прогоняются. Ollama в тестах мокается.
